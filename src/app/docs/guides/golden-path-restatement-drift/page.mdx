# Golden Path: Detect Restatement Drift in a Backtest

This guide demonstrates how financial restatements can alter historical research results and how Arche prevents this issue.

## The Problem

Many financial datasets overwrite historical values when companies restate financial statements.

This creates a subtle issue:

A backtest run today may use data that **did not exist at the time of the historical decision**.

This introduces **look-ahead bias**.

## Scenario

Suppose a strategy screens companies based on net income.

Threshold rule:

```text
Net Income > 96.9B
```

Now consider Apple's fiscal year statement.

## Step 1 - Retrieve Original Statement

Query statements as known on January 1, 2024.

```http
GET /v1/companies/{company_id}/statements?as_of=2024-01-01
```

Example result:

```json
{
  "items": [
    {
      "statement_date": "2023-09-30",
      "net_income": "96995000000",
      "version": 1,
      "is_restated": false
    }
  ]
}
```

Net income: **96.995B**

The company passes the strategy filter.

## Step 2 - Retrieve Restated Statement

Later, the company files an amended statement.

Query the same period again:

```http
GET /v1/companies/{company_id}/statements?as_of=2024-06-01
```

Example result:

```json
{
  "items": [
    {
      "statement_date": "2023-09-30",
      "net_income": "96810000000",
      "version": 2,
      "is_restated": true
    }
  ]
}
```

Net income: **96.81B**

Now the company fails the filter.

## Result

| Dataset Source  | Net Income | Strategy Outcome |
| --------------- | ---------- | ---------------- |
| Original filing | 96.995B    | Included         |
| Restated value  | 96.81B     | Excluded         |

The portfolio changes even though the historical date is identical.

## How Arche Solves This

Arche preserves every statement version.

You can:

- retrieve the original value
- inspect restatement lineage
- reconstruct historical datasets exactly

## Inspect Version Lineage

```http
GET /v1/statements/{statement_id}/versions
```

Response:

```json
{
  "versions": [
    {
      "version": 1,
      "accepted_at": "2023-11-03T20:31:00Z"
    },
    {
      "version": 2,
      "accepted_at": "2024-02-12T21:02:00Z",
      "is_restated": true
    }
  ]
}
```

## Correct Backtest Workflow

1. Select a rebalance date
2. Query statements using `as_of=<rebalance date>`
3. Join statements with market data
4. Run the model

Because Arche preserves version history, results are reproducible.

## Takeaway

Most vendors expose the **latest corrected history**.

Arche exposes the **history that was actually knowable at the time**.
